Searching "Private Ryan", Substring Search in PSQL.

To list rows, where any field contains the search query as a substring.

  • We needed substring search across a huge transactions table. For context, one of our client's DB has nearly 80M rows in a single month's partition. This write-up walks through the thought process behind the implementation, the approaches we considered, and the tradeoffs between storage, query speed, and efficiency.

My experience with SQL was mostly around Data Manipulation, peeking from the application layer through huge abstractions of ORM. I had barely peeled a single layer below SELECT .. FROM .. WHERE. My weapon of choice initially was naïve LIKE operator, which worked just fine with my 10-row synthetically seeded Orders table for my fifth semester e-commerce project. ..WHERE product_name LIKE '%search_query%'.

Side note: Ended up with LIKE anyways.

Before moving forward, let me introduces our distinguished guest tsvector.

Initially, the search function was powered by an extra column called searchable, which stored a tsvector value. In PostgreSQL, a tsvector is a normalized, storage-efficient collection of lexemes designed for full-text search. The column was queried using tsquery, allowing PostgreSQL to match searched tokens against the stored lexemes.

For example:

SELECT to_tsvector(
  'english',
  'This Ryan better be worth it. He''d better go home and cure some disease or invent a longer-lasting lightbulb or something.'
);

Output:

'better':3,8 'cure':12 'diseas':14 'go':9 'hed':7 'home':10 'invent':16 'last':20 'lightbulb':21 'longer':19 'longer-last':18 'ryan':2 'someth':23 'worth':5

Here is what happened:

  • Stop words were removed: words like This, be, it, and, some, or, and a were ignored because they are common English stop words.

  • Words were normalized/stemmed: disease became diseas, lasting became last, and something became someth.

  • Repeated lexemes were stored once with multiple positions: better appears twice, so it is stored once as 'better':3,8.

This is exactly what full-text search is good at: turning natural language into searchable tokens. But our data was seldom a natural language.

tsvector operator is a hugely celebrated feature for text search because with proper indexing, searching even in huge documents becomes sufficiently fast. If your users frequently look for the token 'Shakira' in Shakespeare's novels, then wohoo, there you go, but our users searched with substrings like 05067 in fields that looked like PSW UPI-7175926-050674899-CTZNNPKA K94210543

here's what tsvector produces for cases like this

SELECT to_tsvector(
  'english',
  'PSW UPI-7175926-050674899-CTZNNPKA K94210543'
);
---
'-050674899':4 '-7175926':3 'ctznnpka':5 'psw':1 'k94210543':6 'upi':2

To get any relevant results with tsquery you had to search with a block of code (like k94210543). But, a lexeme lost its meaning when used with machine-generated-identifiers. This approach even with the smartest tokenization, was quickly deemed unhelpful and had to be refactored for substring matching. Oh !! LIKE, I am coming for you.

Before pointing fingers at LIKE, which now is a good friend of mine, I had to hear its side of the story. If you're a C whiz-kid, here's the relevant source code.

How does LIKE work?

Conceptually, PostgreSQL’s LIKE implementation scans the input string and pattern using separate pointers. Literal characters are compared directly, _ advances by one character, and % triggers scanning and recursive attempts against the remaining pattern.

With my then knowledge, I would have written something like,

.... WHERE description like '%search_query%'

So simple, woof, am I a genius? Dunning Kruger can whoop my ass, I assumed my seniors never thought of it. Turned out, my grandpa walked faster than LIKE.

Stumbling upon pg_trgm (The sauce)

LIKE itself was never the problem, the bottleneck was what came in and around it, the search query, the stored text, and mostly the number of candidate rows it had to inspect.

With our previous LIKE query, the inputs were search patterns that had to be matched against raw text values from the candidate rows. But, what if we could change how the stored text and the search query were represented before matching them? Enter trigram, a group of three consecutive characters taken from a string.

Here's an example:

select show_trgm('This Ryan better be worth it. He''d better go home and cure some disease or invent a longer-lasting lightbulb or something.');

OUTPUT:

{"  a","  b","  c","  d","  g","  h","  i","  l","  o","  r","  s","  t","  w"," a "," an"," be"," cu"," d "," di"," go"," he"," ho"," in"," it"," la"," li"," lo"," or"," ry"," so"," th"," wo","an ",and,ase,ast,"be ",bet,bul,cur,dis,eas,ent,"er ",eth,ett,ger,ght,"go ","he ",hin,his,hom,htb,igh,ing,inv,"is ",ise,"it ",las,"lb ",lig,lon,"me ",met,"nd ","ng ",nge,"nt ",nve,ome,ong,"or ",ort,"re ",rth,rya,"se ",sea,som,sti,tbu,ter,"th ",thi,tin,tte,ulb,ure,ven,wor,yan}

As evident, with faster searching comes greater storage requirement. Roughly speaking, with trigrams, the candidate rows can be filtered early by the number of trigrams it shares with the search query, before applying the actual matching operator.

Guess, which operator trigrams support for text searching? No, don't skip ahead, take a moment, wait, take one more second, ummmmm, yup!! the old, cute LIKE operator, a good friend of mine btw. But, this time it came with a trigram armor around it. Now, my friend is suited up for a funeral of it's haters.

Now, I had a presentable strategy for the refactor, use pg_trgm with GiST index. Not much, but honest work. Anyways, with my trust-me-bro benchmark, I concluded 3GB of index for 15M rows to get substring matching in reasonable time was a no-brainer. So, I started bringing the refactor in the application layer.

The Sauce turned sour

To get to this point, I had read enough PostgreSQL documentation and relevant articles to understand the character of indexing. So, while bringing my refactor to the application layer, I (with the help of my senior) realized that all the queries to the table had filters that drastically decreased the scope of the search. For example, a transaction is always scoped to the source, type of it, so naturally only transactions from that scope was a relevant result for the user.

If you have tried devising an indexing strategy ever, you know this is a perfect opportunity for a B-Tree composite index to shine. My trust-me-bro benchmarks showed a 10x faster query results, across the software, with our newest, super-affordable index. It only cost me 500MB in the table while the trigram index was consuming 10GB. Now, coming back to our search, something weird started happening that I couldn't get my head around, no matter how I tweaked my queries or my trigram index, the planner would rarely use our trigram index.

I grew up in a household where shampoo bottles were refilled with water, so my heart shed a tear when my 10GB per-partition index was deemed useless by the planner. Ouch!! still hurts.

The query planner, instead of our trigram index, applied B-tree index scan and used LIKE '%query%' predicate as a filter over those candidate rows.

The actual benefit of the trigram index was candidate reduction. Instead of applying ILIKE '%query%' against every raw text value, PostgreSQL could use trigrams from the search query to find rows that were likely to match, then recheck the actual ILIKE condition. It did so because the estimated cost to

  • first, scan the index,
  • then, combine on other predicates to filter,
  • finally, fetch/recheck those rows never matched the benefit.

In other words, the B-tree index didn't help for substring search but it could cheaply find the small set of rows matching other fields. After that, applying LIKE as a filter was cheap enough.

hmm, back at square zero. But still, I kept the trigram index, with a hope someday it would prove its worth.

A month passed, no complaints, no praises either but that's okay. Then one day, I push through office door, a gust of wind whips my jacket. I narrow my eyes, start steps towards my seat, until a wave of breeze(internet) brings an envelope signed to Shashwat (a Slack message) steals my attention. A complaint, the search has been performing poorly obstructing daily work.

The plan was perfect, the stage was set for my strategy to shine, until another EXPLAIN ANALYZE query showed me my true colors. My trigram index didn't just come with storage constraints, it blew up the entire search function.

Here's what happened, the PostgreSQL planner, which was actively avoiding the trigram index altogether was now deciding to load both indexes, and perform bitmap AND operation. For debugging purpose, I dropped the trigram index altogether and ran the search query. The performance boost was, urmmm, so much better, how much better? slightly, but by how much? 65x better. Yes, same query, when powered with a trigram index had execution time of 1.5 MINUTES, was dropped to 1.5 seconds with NO trigram index.

The trigram index, as designed, was selective in output but expensive in access. It produced only 4 candidate rows, but getting those 4 rows from the 10 GB trigram index took around 103 seconds.

Now, even without the trigram index, my LIKE, best friend btw, backed search runs faster than my grandpa who is on a wheelchair with two rocket boosters, all because it has a strong B-Tree companion which does all the heavy lifting.

Conclusion

The final fix was not a smarter substring-search algorithm. It was a better understanding of the workload.

At first, pg_trgm looked like the obvious answer because the requirement was substring search. PostgreSQL does support trigram indexes for LIKE and ILIKE, and on paper that matched the problem perfectly. But our production query was “search this text inside a specific transaction scope.”

That distinction provided a fresh outlook to the indexing strategy.

The lesson I took from this is simple: do not index the operator in isolation. Index the access pattern.

For this workload, the winning plan was:

  1. use a composite B-tree index to reach the relevant business scope;
  2. apply substring matching only inside that narrowed scope;

  • The scoped B-tree index returned around 784,149 candidate rows.
  • The trigram index returned only 4 candidate rows, but the trigram index scan itself took about 103 seconds.
  • PostgreSQL combined both indexes using BitmapAnd.
  • Dropping the trigram index forced PostgreSQL into the simpler B-tree index scan.
  • Filtering 784,148 rejected rows still took only about 1.58 seconds, which was cheaper than touching the trigram index.
